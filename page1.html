Algorithm

Introduction

Many times we will encounter a challenge or problem. An algorithm is a coherent, step by step solution. They work by recieving an input (or multiple inputs) and preforming a process or operations on said inputs to produce the desired output(s). Instructions need to be coherent to allow a person or computer to preform the correct steps when following an algorithm. Inputs of an algorithm have to be relevent or the algorithm won't be able to preform the logical steps to yeild the correct solution.

Processes and Operations
Definitions

Proccess: A series of operations
Operation: A step required in order to accomplish a specific task

Processes and operations have a one-to-many relationship. One process can consist of many intructions.

Examples
Example 1

Jane wants to eat a cheese sandwich. She does this by taking a slice of chesse, inserting it between two slices of bread and cutting her sandwich diagonally. Jane asks her friend Alice to make her a cheese sandwich. Alice has never made a cheese sandwich before and so asks Jane for instructions.

"Get two slices of bread and one slice of cheese"
"Put the slice of cheese between the two slices of bread"
"Finally slice the sandwich in half diagonally"

In this example, the two slices of bread and slice of cheese are the inputs, the operations involved were putting the slice of cheese between the two slices of bread and slicing the sandwich in half diagonally and the output from this algorithm is Janes version of a cheese sandwich. Something to keep in mind is that many algorithms can produce the same result by using different inputs and processes/ operations (like if the input was an already prepared cheese sandwich and the process carried out was just slicing it diagonally). Another thing to note is the Jane was not very specific about how to cut the sandwich. This is because she infers that Alice already has assumed knowledge of how to cut a sandwich in half, how diagonal she wanted the sandwich to be cut, what she would need to cut the sandwich, etc. One last thing to mention is that what Jane sees as a "Cheese Sandwich" will not always correlate to somebody else's version of a cheese sandwich. You might make a cheese sandwich completely different to how Jane makes hers.

It is important to:

  • The inputs of an algorithm must match the inputs of the algorithm
  • Make sure that the process/ operations are clear and understandable so they can be followed
  • Have a clear idea of the output you want the alorithm to produce

    • Example 2

    A machine takes integers, multiplies them by 5 and outputs the result.

    In this example, the integers are the input, the multiplication by 5 is the operation and the result of the multiplication is the output. You wouldn't input a string or character as the the input doesn't match the input required and you can't multiply words or letters.

    • nodes, which are indicated as ∙ in the picture and
    • edges, which are the connections between the nodes.

    Our pictures suggest that we can treat all the nodes as the same somehow since they are all marked by ∙, but any description of a graph will have to have distinct names for the nodes, and we find it useful to think of them as forming a set.

    We may have even more data to include in our picture, and we talk about that after looking at formal definitions.

    Directed Graphs
    We have to make a distinction between the situation where the edges are directed and that where they work both ways. If there is an edge from a node X to a node Y we say that x is a direct neighbour of y.
    Definition 1: direct graph

    A directed graph is given by a set N of nodes and a set

    E ⊆ ( N x N) \ {(n, n) | n ∈ N}

    of edges.

    In the above definition, we say that the edges in E are pairs of nodes (via E ⊆ ( N x N)) and we exclude edges from a node to itself (via ∖{(n, n) | n ∈ N}). Based on this definition, the following defines a directed graph:

    N = {a, b, c, d, e},

    E = {(a, b), (b, a), (b, c) , (c, b), (d, a), (d, b)}

    We may picture this graph as shown in Figure 2. The example should make it clear that we interpret an edge of the form (x, y) as a connection from x to y. This leaves us with explaining why we are excluding edges of the form (n, n) where n ∈ N. This would be an edge that goes from a node to itself

    Figure 2: A directed grap

    This is not allowed according to our definition. If we wanted this to be a feature we would have to talk about a directed graph with loops but we will not use those in this course unit.

    Also note that by saying that the set of edges forms a subset of the set N × N we ensure that there is at most one edge from a node x to a node y.

    There may be reasons you would like to allow there to be more than one edge, in which case we would speak of a (directed) graph with multi-edges. For this we need to change the formal definition, but we will not be concerned with that possibility in this unit so we do not look into the details.

    Undirected graphs

    What is the difference between a directed and an undirected graph? We think of all the connections as working both ways, but what does that mean for our formal definition?

    There are two solutions one might adopt here. We might either still use a set E ⊆ N × N to give the edges, but demand that (x, y) ∈ E implies (y, x) ∈ E, to ensure that all edges go both ways. But that would make the set E twice as big as it needs to be.

    Instead we use 2-element subsets of N to describe the edges of our graph, making use of one of the features of sets, namely that the order of elements does not matter.

    Definition 2: undirect graph

    An undirected graph is given by a set N of nodes and a set

    E ⊆ {{x,y} | x,y ∈ N, x is not equal to y}

    of edges.

    If {x, y} ∈ E, we say that x and y are neighbours, i.e., x is a neighbour of y and y is a neighbour of x.

    Again we forbid loops by not allowing one-element sets of the form {x} to define an edge (which would have to be from x to itself). We again do not allow multi-edges in our definition since we demand that all the edges form a set, which rules out duplicates. We may now consider the graph

    N = {a, b, c, d, e},

    E = {(a, b), (b, a), (b, c) , (c, b), (d, a), (d, b)}

    which we may picture as shown in Figure 3.

    Complexity

    Introduction

    Complexity is the measure the effort it takes to run on some input. It uses the given size of the input, how much time it takes and how much space it takes and uses the worst case senario. It is denoted using O notation and looks like "O(n)" where n is the amount of time or space it would use.

    Definitions

    Complexity: Measure the effort it takes to run on some input

    O Notation

    To calculate the time complexity of an algorithm, you must count how many steps it would take in a worst case senario:

    Simple instructions (e.g. print, read): 1 Step (O(n))
    If statement: However many steps are in the if statement
    If Else statement: Whichever has the most steps
    If Elif Else statement: Whichever has the most steps
    Loops: The number of steps in the loop multiplied by the number of iterations (O(n*i) where i is the iterations)
    Nested Loops: The product of each loops iterations (O(n^j) where j is the number of nested loops)

    Theses are to be added together (using basic BIDMAS) to produce the time complexity of an algorithm (e.g. O((n^3)+4))
